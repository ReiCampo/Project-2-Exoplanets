Best Fit Analysis¶

Now that we have an idea of what the data is like, we will define functions for calculating the luminosity distance and distance modulus, respectively, which are defined for the two main models we are constraining the parameters of:

  • CDM: A flat cosmological model that shows the current state of our universe today with the accompanying cosmological parameters. According to Sclonic et al. (2018), this means that our universe is "flat, accelerating, and primarily composed of baryons, dark matter, and dark energy." There is the radiation density, but we exclude it because we do not use it in this analysis. The cosmological pararmeters we are attemtping to constrain for this model are M, the matter density parameter, and , the dark energy density parameter.

  • CDM: A flat cosmological model that describes a universe where the dark energy is dictated by how the matter density changes. The cosmological parameters we are attempting to constraing for this model are M, again the matter density parameter, and , which in this model is a constant equation of state parameter for dark energy equivalent to P/, where P is the pressure and is the energy density of dark energy.

Note: As one can see, the Hubble constant is set to a value of 70 km/s Mpc. While this is indeed an experimentally determined value that would ideally be constrained as part of our model, it is difficult to do so given our method of analysis and therefore it remains as the experimentally accepted approximate value for our purposes.

def dL_astropy_lambda( M, , z, w ):
    ## Calculates the distance (dL) using astropy and returns the values as an array in units of pc
    dL = LambdaCDM(H0=70, Om0=M, Ode0=)
    return dL.luminosity_distance(z).to_value(u.pc)

def _model_lambda(M, , z, w ):
    ## (Note: it's  (written as '\mu' then tab), not u ) Finds a model distance in using redshift
    ## using redshift and cosmologial parameters.
     = 5 * np.log10(dL_astropy_lambda(M, , z, w)/10)
    return 

def dL_astropy_omega( M, , z, w ):
    ## Similar function to dL_astropy_lambda, only the astropy model is changed to the wCDM model
    dL = wCDM(H0=70, Om0=M, Ode0=,w0=w)
    return dL.luminosity_distance(z).to_value(u.pc)

def _model_omega( M, , z, w ):
    ## Operates the same as _model_lambda but makes a call to dL_astropy_omega instead
     = 5 * np.log10(dL_astropy_omega( M, , z, w )/10)
    return

We now arrive at the point where we define the method in which we constrain the parameters mentioned above. In order to constrain the parameters, we define a penalty function, known as $\chi^{2}$ or the "chi square" function, that, in this case, takes the form

$ \chi^2(m,d) = \sum_{i=1}^{n} \left( \frac{m_{i}-d_{i}}{\sigma_{i}} \right)^2 $

where m is some function with parameters yet to be determined, d is an experimentally determined set of values for that function, $\sigma$ is the error of those experimentally determined values, and n is the number of data points in d. Minimizing $\chi^{2}$ by varying the values of the parameters of m allows one to determine the values of these parameters so they best fit the experimental data. We accomplish this using the scipy library's $\texttt{optimize.fmin}$ function, which returns the best fit for the parameters as well as the number of iterations it went through to determine the current value and the number of times the function was evaluated. In this case, the output of the function is supressed so these attributes aren't printed every time the function is called, which will be convenient later.

Since $\texttt{optimize.fmin}$ requires an initial guess for the parameters, one runs into the dilemna of the potential bias derived by the initial guess for the magnitude. One could just use the first sample, but that could result in a slight vasriation in the best fit considering how biased the data is towards low redshift data points. To reduce this issue, we created functions that determine the average absolute magnitude for each model given the entire data set using the same idealized approximate values for $\Omega_{M}$ and $\Omega_{\Lambda}$ that we use as the initial guess for those values. While there is still a systemic bias due to the data set, this reduces the bias of our analysis of the best fit.

Markov Chain Monte Carlo (MCMC) Analysis and Parameter Uncertainty¶

Since we know the data set itself has a standard error, it follows that there is a degree of error in the best fit we derived. At first glance, the range of plausible values seems very difficult to determine, but fortunately there exists an oppurtunity to find this range in the form of Markov Chain Monte Carlo (MCMC) analysis. To put it simply, MCMC analysis takes a lightly constrained probability function (in our case, we constrain the parameters to ensure they are of the correct sign and don't vastly exceed the amount of matter in the universe) and traces out an area of plausible values for this function.

To do this, one has a set of "walkers" start at a given value for each parameter being evaluated and has them take a very small step in a random direction. With enough steps, the walkers will trace out a range of plausible values for the parameters. In order to avoid a dependence on the starting position of each value, it is suggested to do a "burn-in" run where a smaller number of steps are taken by the walkers before saving their random position at the end of the burn-in run and resetting the algorithm. From there, one uses these randomized intitial conditions to then do a longer testing run, where the walkers will trace out the plausible values. While increasing the number of walkers and the number of steps increases the accuracy of this range of values, MCMC analysis can quickly become extremely computationally expensive, especially in our case, where we do not make use of multithreading. As a result, we have used the highest number of walkers and steps we can have run in a reasonable amount of time.

In order to accomplish this, we make use of the $\texttt{emcee}$ python library, which allows one to conduct MCMC analysis much quicker than creating the model from scratch. In our case, the probability function is characterized by use of the previously explained penalty function alongside a small number of constraints to keep the MCMC analysis from trying to evaluate values of the parameters that would return errors. Each parameter requires its own initial condition for both the burn-in and main run. In this instance, we are analysing the $\Lambda$CDM model, so we plot the values of $\Omega_{M}$ vs. $\Omega_{\Lambda}$ to see the generally plausible values. While this is a useful visualization of the errors of the parameters, it doesn't tell us concretely what the error values are, nor does it plot all of the parameters together. Fortunately, the $\texttt{corner}$ library allows us to do just that, and, using the values for each parameter in the array created by $\texttt{emcee}$, we can see the distribution of the values of each parameter vs each other, a histogram of the values of each parameter, and values for the uncertainty of each parameter.
